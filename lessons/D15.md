## Part 1: The Concepts

### From single-shot to multi-step: the eval surface expands

Every lesson so far assumed the AI system produces one output per input. You give it a menu description; it returns approve, reject, or flag. The eval compares that one output to ground truth.

Agents break this assumption. An agent doesn't just respond — it **acts**. It plans, calls tools, observes results, and decides what to do next. One user request can generate a sequence of 3, 7, or 15 steps before the agent is done.

This expands the eval surface dramatically. In a single-shot system, you evaluate one output. In an agent, every step is a potential eval target: Did the agent choose the right tool? Did it call it with the right arguments? Did it interpret the tool output correctly? Did it make the right decision about what to do next? The number of things that can go wrong scales with the number of steps — and so does the number of things you have to measure.

Evaluating only the final output hides everything that happened along the way. An agent can reach the right answer through a disaster of a process, and you'd never know.

### Outcome vs trajectory: two dimensions of correctness

Agent evaluation has two dimensions, not one.

**Outcome evaluation** asks: did the agent complete the task correctly? Final answer right or wrong. This is what traditional eval looks like — a pass/fail on the end state.

**Trajectory evaluation** asks: did the agent take a reasonable path to get there? Did it use tools appropriately? Did it follow a sensible plan? Were any steps redundant, incorrect, or dangerous?

Most teams evaluate only outcome. That hides four failure modes that will catch up with you at scale:

- **Lucky correctness** — the agent stumbled onto the right answer through random steps. Works for one case; fails at scale.
- **Hidden cost** — the agent took 15 steps when 3 would have sufficed. Each extra step is tokens, latency, and money.
- **Hidden risk** — the agent called destructive tools (sent emails, modified records) that happened not to cause harm this time. Next time might be different.
- **Hallucinated steps** — the agent invented tool calls or tool results that don't exist. The answer is built on fabricated scaffolding.

A correct outcome through a bad trajectory is a **deferred failure**. It works until the random noise lines up wrong.

### Tool use is a distinct failure mode

When an agent calls tools, there are four places it can fail, and each requires different remediation:

1. **Wrong tool selected** — agent reaches for the wrong capability. Fix: better tool descriptions, few-shot examples, stricter planning prompts.
2. **Right tool, wrong arguments** — agent picked the correct tool but malformed the call. Fix: stricter schema validation, type hints, argument examples.
3. **Right tool, right arguments, wrong result interpretation** — the tool returned correctly; the agent misread what it meant. Fix: prompt the agent to summarize tool outputs before acting on them.
4. **Correct tool use, wrong next step** — everything was right until the agent decided what to do next. Fix: planning prompt improvements.

You can't diagnose which of these is happening unless you evaluate tool use as its own dimension. Collapsing them all into "the agent failed" tells you nothing about how to fix it.

### Error recovery separates resilient agents from brittle ones

Agents operate in a messy world. Tools time out. APIs return 500s. Search returns no results. Responses don't match expected schemas. Errors aren't edge cases — they're routine.

A resilient agent notices the error, adapts, and finds an alternative path. A brittle agent cascades: the failed step produces garbage output, the agent treats it as valid, the next step builds on nonsense, and the trajectory spirals.

Evaluating error recovery is as important as evaluating happy paths. The key questions:
- When a tool fails, does the agent detect it?
- Does it retry, try an alternative, or give up cleanly?
- If it gives up, does it explain the failure to the user — or hallucinate an answer to avoid admitting it couldn't finish?

Cascade failures are among the most expensive failure modes in agent systems. They burn tokens, produce confidently wrong outputs, and are invisible to outcome-only evaluation.

### Efficiency is a first-class metric for agents

For single-shot systems, cost and latency are optimization concerns. For agents, they are guardrails.

A single LLM call costs X tokens. A 10-step agent costs 10X plus tool overhead plus re-read context. An agent that takes 12 steps when 3 would suffice costs 4x more than one that plans well — for the same user outcome.

Step bloat is a silent failure mode that outcome evaluation doesn't catch. The agent "works," but each user interaction burns unnecessary money. At scale, this breaks the unit economics of the product.

The PM discipline: track **steps per successful task** as a core metric, set a budget per task type, and treat step bloat as a regression, not an efficiency opportunity. Agent systems that aren't measured on efficiency drift toward wastefulness because LLMs are biased toward verbosity.

## Part 2: Exercise — Evaluating a Menu Verification Agent

### Scenario

Engineering has built the first version of a **menu verification agent**. Unlike the single-shot classifier you've evaluated so far, this agent can take actions: it queries the photo upload API, cross-references the allergen database, looks up restaurant history, escalates borderline cases to human review, and emails restaurants for clarification.

The agent has been running in shadow mode for two weeks. You have 25 sample runs logged with trajectory details. Before the team proposes moving the agent to production, your job is to evaluate not just whether it got the right answer — but *how* it got there.

### Dataset

Open `exercises/agent-trajectories-dataset.csv`. Each row is one agent run.

| Column | What it means |
|--------|---------------|
| `run_id` | Identifier (R-01 through R-25) |
| `task_type` | Category of task: new_menu_upload, allergen_dispute, photo_verification, description_update |
| `outcome_correct` | Did the agent reach the right final decision? (yes/no, based on post-hoc human review) |
| `trajectory_quality` | Process classification: efficient, recovered_from_error, inefficient, cascade_failure, hallucinated_tool_call |
| `num_steps` | How many steps the agent took (1–15) |
| `expected_steps` | Minimum steps a well-functioning agent would take for this task (expert-labeled ground truth) |
| `wrong_tool_calls` | Count of steps where the agent called the wrong tool or used wrong arguments |
| `tool_failure_count` | Count of tool calls that returned errors during the run |
| `recovered_from_failures` | Count of tool failures the agent successfully recovered from (by retry or alternative) |
| `cost_tokens` | Total tokens consumed across all steps |

### Expected outcome

By the end, you'll have: (1) computed outcome accuracy vs trajectory quality and seen how they diverge, (2) diagnosed the dominant trajectory failure mode, (3) quantified step efficiency and found where step bloat is concentrated, (4) written an eval spec that closes the loop from a dangerous trajectory failure back to the regression suite.

### Step 1: Compute outcome accuracy vs trajectory quality

Count: How many runs had `outcome_correct = yes`? How many had `trajectory_quality` in one of the "good" categories (efficient or recovered_from_error)? How do those two numbers compare?

Then identify the runs where `outcome_correct = yes` but `trajectory_quality` is one of the bad categories (inefficient, cascade_failure, hallucinated_tool_call). These are **lucky correctness** cases — right answer, wrong path. They're your hidden risk surface: outcome-only evaluation would miss them entirely.

### Step 2: Diagnose the dominant trajectory failure mode

Among runs with bad trajectory quality, what's the most common pattern? Count inefficient, cascade_failure, and hallucinated_tool_call occurrences. Cross-reference with `wrong_tool_calls` and `tool_failure_count` to see whether bad trajectories are dominated by tool-use errors, error-recovery failures, or step bloat.

This matters because each has a different remediation path (from Concept 3). Knowing which failure mode dominates tells you what to fix first.

### Step 3: Quantify step efficiency

For each run, compute step_ratio = `num_steps` / `expected_steps`. What's the median? How many runs have a ratio ≥ 2x (taking twice or more as many steps as needed)? Is the step bloat concentrated in a specific `task_type`?

This is the cost discipline from Concept 5 applied to real data.

### Step 4: Close the loop from observability to evaluation

Look at the runs where `trajectory_quality = hallucinated_tool_call`. These are the most dangerous cases in the dataset — the agent produced a correct-looking output built on fabricated tool calls that never actually happened.

Pick one of those runs. Write an eval spec (D13 format: input, expected output, pass criteria, failure mode area, severity) that would catch this failure mode *before* the agent reaches production. The spec should measure **trajectory quality**, not just outcome — meaning the pass criteria must include something about the tool-call log, not just the final decision.

## Part 3: PM Decision Point

Based on your analysis, you're preparing a ship recommendation. Your engineering lead pushes:

*"Outcome accuracy is 20/25 = 80%. We're within the 75% threshold we set for shadow testing. Let's ship it and iterate on trajectory issues after GA."*

Push back — or don't — using your analysis. If you believe the outcome-only framing is wrong, name the specific trajectory failure modes that make it unsafe to ship right now, and the specific investments you'd make before GA. If you believe the agent is ready, explain why the trajectory issues you found are acceptable risk.

Either way, your recommendation must cite specific numbers from your analysis: outcome accuracy, lucky-correctness count, dominant failure pattern, step efficiency, and the trajectory types you'd block ship on.
